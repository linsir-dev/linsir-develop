## 核心概念

### CMM核心概念

#### 1. 能力成熟度模型

CMM（Capability Maturity Model，能力成熟度模型）是由美国卡内基梅隆大学软件工程研究所（SEI）在20世纪80年代末开发的一种软件过程改进模型。CMM旨在帮助软件开发组织改进其软件开发过程，提高软件开发的质量和生产效率。

#### 2. 五个成熟度级别

CMM定义了五个成熟度级别：

- **初始级（Level 1: Initial）**：软件开发过程是混乱的，缺乏一致性和可预测性
- **可重复级（Level 2: Repeatable）**：组织已经建立了基本的项目管理过程，可以重复以前成功项目的实践
- **已定义级（Level 3: Defined）**：组织已经建立了标准的软件开发过程，所有项目都使用这个标准过程
- **已管理级（Level 4: Managed）**：组织已经建立了度量和分析机制，可以定量地管理和控制软件开发过程
- **优化级（Level 5: Optimizing）**：组织已经建立了持续改进的机制，可以持续改进软件开发过程

#### 3. 18个关键过程域

CMM定义了18个关键过程域（Key Process Areas，KPAs）：

- **可重复级（6个）**：需求管理、项目计划、项目跟踪、子合同管理、质量保证、配置管理
- **已定义级（7个）**：组织过程焦点、组织过程定义、培训计划、集成软件管理、软件产品工程、组间协调、同行评审
- **已管理级（2个）**：定量过程管理、软件质量管理
- **优化级（3个）**：缺陷预防、技术革新、过程变更管理

### CMMI核心概念

#### 1. 能力成熟度模型集成

CMMI（Capability Maturity Model Integration，能力成熟度模型集成）是由美国卡内基梅隆大学软件工程研究所（SEI）在2000年发布的一种能力成熟度模型集成。CMMI集成了多个能力成熟度模型，包括软件工程（SW-CMM）、系统工程（SE-CMM）、集成产品开发（IPD-CMM）等，扩展了应用领域，提高了灵活性。

#### 2. 五个成熟度级别

CMMI定义了五个成熟度级别：

- **初始级（Level 1: Initial）**：过程是混乱的，缺乏一致性和可预测性
- **已管理级（Level 2: Managed）**：组织已经建立了基本的项目管理过程，可以重复以前成功项目的实践
- **已定义级（Level 3: Defined）**：组织已经建立了标准的过程，所有项目都使用这个标准过程
- **量化管理级（Level 4: Quantitatively Managed）**：组织已经建立了度量和分析机制，可以定量地管理和控制过程
- **优化级（Level 5: Optimizing）**：组织已经建立了持续改进的机制，可以持续改进过程

#### 3. 22个过程域

CMMI定义了22个过程域（Process Areas，PAs）：

- **已管理级（7个）**：需求管理、项目计划、项目监控、供应商协议管理、度量与分析、过程和产品质量保证、配置管理
- **已定义级（11个）**：需求开发、技术解决方案、产品集成、验证、确认、组织过程焦点、组织过程定义、组织培训、集成项目管理、风险管理、决策分析与解决
- **量化管理级（2个）**：组织过程性能、定量项目管理
- **优化级（2个）**：组织创新与部署、原因分析与解决

#### 4. 两种表示方法

CMMI提供了两种表示方法：

- **阶段式表示法（Staged Representation）**：类似于CMM，将过程域按成熟度级别分组
- **连续式表示法（Continuous Representation）**：将过程域按过程能力级别分组，提供更大的灵活性

### ISO9000核心概念

#### 1. 质量管理体系

ISO 9000是由国际标准化组织（ISO）制定的质量管理体系标准，包括ISO 9000、ISO 9001、ISO 9004等标准。ISO 9000旨在帮助组织建立、实施、维护和改进质量管理体系，提高产品和服务的质量，增强客户满意度。

#### 2. 八个质量管理原则

ISO 9000基于八个质量管理原则：

1. **客户导向**：组织依存于其客户，因此应理解客户当前和未来的需求，满足客户要求，并争取超越客户期望
2. **领导作用**：领导者确立组织统一的宗旨和方向，他们应该创造并保持使员工能充分参与实现组织目标的内部环境
3. **全员参与**：各级人员都是组织之本，只有他们的充分参与，才能使他们的才干为组织带来收益
4. **过程方法**：将活动和相关的资源作为过程进行管理，可以更高效地得到期望的结果
5. **系统方法**：将相互关联的过程作为系统来看待、理解和管理，有助于组织提高实现目标的有效性和效率
6. **持续改进**：持续改进总体业绩应当是组织的一个永恒目标
7. **基于事实的决策方法**：有效决策是建立在数据和信息分析的基础上
8. **互利的供方关系**：组织与供方是相互依存的，互利的关系可增强双方创造价值的能力

## 对比分析

### CMMI与CMM的对比

#### 主要区别

| 对比维度 | CMM | CMMI |
|---------|-----|------|
| 定义 | 能力成熟度模型 | 能力成熟度模型集成 |
| 发布时间 | 1987年 | 2000年 |
| 应用领域 | 主要应用于软件开发 | 扩展到系统工程、集成产品开发、服务管理等多个领域 |
| 模型结构 | 五级成熟度模型，18个关键过程域 | 五级成熟度模型，22个过程域 |
| 表示方法 | 只有阶段式表示法 | 提供阶段式和连续式两种表示法 |
| 评估方法 | CBA-IPI、SCAMPI | SCAMPI A类、B类、C类评估 |
| 灵活性 | 灵活性较低 | 灵活性较高 |
| 兼容性 | 与ISO 9001等标准的兼容性较差 | 与ISO 9001等标准有更好的兼容性 |

#### 相同点

- **五级成熟度模型**：CMM和CMMI都是五级成熟度模型
- **过程改进**：CMM和CMMI都旨在帮助组织改进过程
- **质量保证**：CMM和CMMI都提高了质量保证的水平
- **项目管理**：CMM和CMMI都改进了项目管理的方法
- **可预测性**：CMM和CMMI都提高了项目结果的可预测性

#### 不同点

- **应用领域**：CMM主要应用于软件开发，CMMI扩展到多个领域
- **模型结构**：CMM有18个关键过程域，CMMI有22个过程域
- **表示方法**：CMM只有阶段式表示法，CMMI提供阶段式和连续式两种表示法
- **评估方法**：CMM的评估方法较少，CMMI的评估方法较多
- **灵活性**：CMM的灵活性较低，CMMI的灵活性较高
- **兼容性**：CMM与其他标准的兼容性较差，CMMI与其他标准的兼容性较好

### CMM与ISO9000的对比

#### 主要区别

| 对比维度 | CMM | ISO9000 |
|---------|-----|---------|
| 定义 | 能力成熟度模型 | 质量管理体系标准 |
| 目标 | 改进软件开发过程 | 建立质量管理体系 |
| 发布机构 | 卡内基梅隆大学软件工程研究所（SEI） | 国际标准化组织（ISO） |
| 发布时间 | 1987年 | 1987年 |
| 应用领域 | 主要应用于软件开发 | 适用于各种行业和领域 |
| 模型结构 | 五级成熟度模型，18个关键过程域 | 质量管理体系原则和要求 |
| 评估方法 | CBA-IPI、SCAMPI | 内部审核、外部审核 |
| 评估结果 | 成熟度级别 | 认证结果 |

#### 相同点

- **质量保证**：CMM和ISO 9000都提高了质量保证的水平
- **过程改进**：CMM和ISO 9000都旨在帮助组织改进过程
- **持续改进**：CMM和ISO 9000都强调持续改进
- **客户导向**：CMM和ISO 9000都强调客户导向

#### 不同点

- **应用领域**：CMM主要应用于软件开发，ISO 9000适用于各种行业和领域
- **模型结构**：CMM是五级成熟度模型，ISO 9000是质量管理体系原则和要求
- **评估方法**：CMM的评估方法与ISO 9000的评估方法不同
- **评估结果**：CMM的评估结果是成熟度级别，ISO 9000的评估结果是认证结果

## 常见问题

### CMM相关

#### Q1: CMM是否适用于所有软件开发组织？

A: CMM适用于大多数软件开发组织，但不是所有软件开发组织。CMM假设软件开发过程可以标准化，但在某些情况下，软件开发过程可能无法标准化，因此CMM可能不适用于某些特殊的软件开发组织。

#### Q2: 如何选择合适的CMM成熟度级别？

A: 根据组织的实际情况和需求选择合适的CMM成熟度级别：
- **初始级**：适用于软件开发过程混乱的组织
- **可重复级**：适用于需要建立基本项目管理过程的组织
- **已定义级**：适用于需要建立标准软件开发过程的组织
- **已管理级**：适用于需要定量管理和控制软件开发过程的组织
- **优化级**：适用于需要持续改进软件开发过程的组织

### CMMI相关

#### Q1: CMMI是否适用于所有组织？

A: CMMI适用于大多数组织，但不是所有组织。CMMI假设过程可以标准化，但在某些情况下，过程可能无法标准化，因此CMMI可能不适用于某些特殊的组织。

#### Q2: 如何选择合适的CMMI表示方法？

A: 根据组织的实际情况和需求选择合适的CMMI表示方法：
- **阶段式表示法**：适用于需要逐步提高成熟度级别的组织
- **连续式表示法**：适用于需要灵活选择过程域的组织

### ISO9000相关

#### Q1: ISO 9000是否适用于所有组织？

A: ISO 9000适用于大多数组织，但不是所有组织。ISO 9000假设组织可以建立质量管理体系，但在某些情况下，组织可能无法建立质量管理体系，因此ISO 9000可能不适用于某些特殊的组织。

#### Q2: 如何通过ISO 9000认证？

A: 通过ISO 9000认证的步骤：
1. **建立质量管理体系**：根据ISO 9001的要求建立质量管理体系
2. **内部审核**：进行内部审核，检查质量管理体系的符合性
3. **外部审核**：请第三方认证机构进行外部审核
4. **获得认证**：通过外部审核后，获得ISO 9000认证

## 总结

CMM（Capability Maturity Model，能力成熟度模型）和CMMI（Capability Maturity Model Integration，能力成熟度模型集成）是软件过程改进的重要模型。CMM是由美国卡内基梅隆大学软件工程研究所（SEI）在20世纪80年代末开发的一种软件过程改进模型，旨在帮助软件开发组织改进其软件开发过程，提高软件开发的质量和生产效率。CMMI是CMM的继承和发展，在CMM的基础上，集成了多个能力成熟度模型，扩展了应用领域，提高了灵活性。ISO 9000是由国际标准化组织（ISO）制定的质量管理体系标准，适用于各种行业和领域。CMM和ISO 9000是两个不同的质量管理体系，在定义、历史背景、应用领域、评估方法、优缺点等方面都有显著的区别。通过学习CMM/CMMI和ISO 9000，可以提高软件过程改进和质量管理的水平，设计出更好的软件质量管理体系。在实际应用中，需要根据实际情况选择应用CMM/CMMI或ISO 9000，或者结合使用CMM/CMMI和ISO 9000，发挥各自的优势，提供更全面的质量管理。
